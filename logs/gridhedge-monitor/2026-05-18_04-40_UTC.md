## Anomalies

### 🆕 A3 — Worker EF Core `DbUpdateException 22001: varchar(20)` — **3 хита за окно, но не от наших ботов**

В `docker compose logs --since=65m worker` зафиксированы три отказа `SaveChangesAsync` с PostgresException 22001 ("value too long for type character varying(20)"):

```
fail: CryptoBotWeb.Worker.TradingHostedService[0]
      Error processing strategy 105dc7c0-68f4-4e33-9c56-66338aca667e (Test3)
      Microsoft.EntityFrameworkCore.DbUpdateException ...
       ---> Npgsql.PostgresException (0x80004005): 22001: value too long for type character varying(20)
```

**Источник:** стратегия `105dc7c0` с именем "Test3" — НЕ из workspace `Grid+Hadge`. Наши S1/S2/S3 в этих стек-трейсах не упомянуты.

**Однако — это latent-баг и в GridHedge коде.** Поле, превысившее лимит, — `Trade.Status varchar(20)` ([backend/src/CryptoBotWeb.Infrastructure/Data/AppDbContext.cs:130](backend/src/CryptoBotWeb.Infrastructure/Data/AppDbContext.cs#L130) `e.Property(x => x.Status).HasMaxLength(20)`). `GridHedgeHandler.RecordTrade` пишет статусы:
- ✓ `MarketEntry@0%` (14) — ok
- ✓ `HedgeOpen` (9), `HedgeClose` (10) — ok
- ✓ `GridFill@-3.5%` (14), `TpFill@-3.5%` (12) — ok
- ❌ **`ForceMarketClose@-3.5%` (22)** — превысит лимит на `CloseEverythingAsync` (stop-loss / upper-exit)
- ❌ **`TpCancelledPartial@-3.5%` (24)** — превысит при cancel+partial-fill TP

Сейчас наши боты в Phase=Active, ни один не близок к exit. Но если **S3** дотянется до upper-exit @ 1.4812 (сейчас 1.3929, нужно +6.34%) или stop-loss @ 1.3402 (нужно −3.78%) — `RecordBatchClosure(reason: "ForceMarketClose", ...)` упадёт с `DbUpdateException` на каждом батче. По коду handler не ловит эту ошибку отдельно (`SaveState`+`SaveChangesAsync` в `finally`-блоке), вся транзакция тика откатится → state не сохранится → trade не запишется. Бот будет крутиться в Phase=ExitingDown/Up в бесконечном цикле падений. **Это критичный риск для финала цикла.**

Рекомендация (не делать в рамках мониторинга, отдать пользователю): расширить колонку `Trade.Status` до `varchar(40)` или сократить статусы в `RecordBatchClosure` (`ForceClose@-X%`).
